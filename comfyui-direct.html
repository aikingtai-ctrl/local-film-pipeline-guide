ComfyUI 还是 SwarmUI?以及 n8n 直连 ComfyUI 的做法

你已经在用 ComfyUI,不一定需要 SwarmUI。本页讲清它俩的关系,并给出 n8n 直连 ComfyUI 的工作流(含轮询分卡),跟你现在的用法对齐。

1它俩不是一个东西,但关系很近

SwarmUI 是套在 ComfyUI 外面的“总管”,底层干活的还是 ComfyUI。

SwarmUI ← 总管:友好界面 + 统一 API + 自动把任务分给多张卡 │ ┌──────┼──────┐ (SwarmUI 自己不生成图,它把活转发给下面的 ComfyUI) ComfyUI ComfyUI ComfyUI ← 真正干活的引擎(你现在用的就是这个) (卡0) (卡1) (卡2)
ComfyUISwarmUI
是什么真正的生成引擎(跑扩散/视频模型)管理器,装的时候把 ComfyUI 打包当后端
干不干活活是它干的自己不生成,只转发+分卡
多卡一个实例一张卡,要多卡就起多个实例自动把任务负载均衡到多个 ComfyUI
你现在✅ 已经在用可选,没装也行
结论:SwarmUI 不是必需品。我当初推荐它,唯一原因是你有 12 张卡、手动管“这镜发哪张卡”很烦。但你也可以让 n8n 自己轮流分派,直连 ComfyUI——少一层复杂度,用你熟的工具。

2两条路,二选一

方案怎么做适合
A. n8n 直连 ComfyUI
(本页,推荐先走)		每卡起一个 ComfyUI 实例,n8n 轮流分派,调 /prompt 提交 + /history 查结果想用熟悉的工具、不想多装一层
B. 加 SwarmUI 当总管
(见进阶部署页)		把多个 ComfyUI 登记进 SwarmUI,n8n 只调 SwarmUI 一个 API12 卡想省心、不想在 n8n 写分卡逻辑
两条路随时能切换、互不冲突。先走 A 跑通,量大了再上 B。下载区给的是 A 的工作流。

3每张卡起一个 ComfyUI 实例

直连方案下,你自己管多卡:一张卡一个 ComfyUI 进程,用 CUDA_VISIBLE_DEVICES 钉卡、各占一个端口。

在有卡的服务器上(ComfyUI 目录内)· 示例起 3 个
for i in 0 1 2; do
  port=$((8188 + i))                       # 卡0→8188 卡1→8189 卡2→8190
  CUDA_VISIBLE_DEVICES=$i python main.py \
    --listen 0.0.0.0 --port $port \
    --output-directory /media &            # 产物落共享盘
done
验证:nvidia-smi 看到每张卡各一个 python 进程;浏览器开 http://IP:8188 能进 ComfyUI 界面。

4ComfyUI 的两个关键接口

n8n 就靠这两个接口驱动 ComfyUI(不用 SwarmUI):

动作接口说明
提交任务POST /promptbody 是你的工作流图 + client_id,返回 { "prompt_id": "..." }
查结果GET /history/{prompt_id}轮询,做完后返回 outputs,里面有产物文件名

关键:先从 ComfyUI 导出“API 格式”工作流

在 ComfyUI 界面把你的出图/生视频流程搭好 → 设置里打开 Enable Dev mode Options → 点 Save (API Format) 导出一个 JSON。这个 JSON 就是 POST /prompt 要发的 prompt 字段内容。n8n 里用一个 Code 节点把「提示词、种子、输入图、输出前缀」注入这个模板即可。

Code 节点注入示例(节点号按你的工作流改)
const graph = JSON.parse(JSON.stringify(TEMPLATE));   // 你导出的 API 格式
graph['6'].inputs.text = `${s.prompt_visual}, ${s.style_global}`;  // 正向提示
graph['3'].inputs.seed = s.seed;                     // KSampler 种子
graph['10'].inputs.image = s.keyframe_path;          // 图生视频的输入图
graph['9'].inputs.filename_prefix = `videos/${s.shot_id}`;  // 输出命名
return [{ json: { endpoint: s.endpoint, body: { prompt: graph, client_id: 'n8n' } } }];

5n8n 里怎么“分卡”(轮询)

没有 SwarmUI 帮你分卡,就让 n8n 自己轮流发:把每张卡的 ComfyUI 地址列成一个数组,按镜头序号取余分配。下面的工作流已内置这段逻辑,你只要把地址改成自己的。

「展开镜头 + 轮询分卡」节点里的关键行
const ENDPOINTS = [
  'http://localhost:8188',   // ← 改成你每张卡的 ComfyUI 地址
  'http://localhost:8189',
  'http://localhost:8190'
];
// 第 idx 个镜头 → ENDPOINTS[idx % 张数],均匀分到各卡

6下载:ComfyUI 直连版工作流

SUB_render_comfyui.json 直连 ComfyUI:展开→轮询分卡→注入模板→/prompt→轮询→回填
1

导入后要改的地方

  • 「展开镜头 + 轮询分卡」:把 ENDPOINTS 改成你每卡的 ComfyUI 地址。
  • 「组装 ComfyUI 工作流」:把 TEMPLATE 换成你导出的 API 格式工作流,节点号(6/3/10/9)按实际改。
  • Redis 节点:填你的 Redis 地址。
  • 回填路径:确认与你 ComfyUI 的 filename_prefix 输出规则一致。
done 计数怎么来?本工作流用轮询 Redis 的 done 计数判断是否全做完。最稳的做法是给 ComfyUI 配一个“任务完成回调”(产物落盘后 INCR 一次);简单起步也可以改成逐个 GET /history/{prompt_id} 判断每镜是否完成再计数。
和 SwarmUI 版的区别:只有“提交”那一步不同——SwarmUI 版调 /API/GenerateText2Image,直连版调 ComfyUI 的 /prompt 并自己轮询分卡。其余(展开、轮询、回填、质检)完全一样。想省心切回 SwarmUI,换掉提交节点即可。
