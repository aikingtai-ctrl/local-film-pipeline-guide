进阶部署:12 卡逐卡配置 + n8n 流水线导入

把 12 张显卡一张张登记进画面工厂,再把搭好的流水线一键导进 n8n。这是主指南第 5.3 / 5.6 步的“保姆级”细化版。

CSwarmUI 12 卡逐卡登记参数表

核心原则:一张卡 = 一个后端 = 一个独立进程,用 CUDA_VISIBLE_DEVICES 把进程钉死在指定卡上,再登记进 SwarmUI。SwarmUI 之后会自动把任务派给空闲后端。

第 1 步:在 A 上拉起 6 个视频后端

服务器 A 终端 · 6 张 H100 各起一个 ComfyUI(在 ComfyUI 目录下)
for i in 2 3 4 5 6 7; do
  port=$((8186 + i))                       # GPU2→8188 … GPU7→8193
  CUDA_VISIBLE_DEVICES=$i python main.py \
    --listen 0.0.0.0 --port $port \
    --output-directory /media &            # 产物落共享盘
done

第 2 步:在 B 上拉起 3 个出图后端

服务器 B 终端 · 3 张 5090 各起一个 ComfyUI
for i in 0 1 2; do
  port=$((8188 + i))                       # GPU0→8188 GPU1→8189 GPU2→8190
  CUDA_VISIBLE_DEVICES=$i python main.py \
    --listen 0.0.0.0 --port $port \
    --output-directory /media &
done

第 3 步:在 SwarmUI 里把这 9 个登记为后端

打开 http://SERVER_A:7801 → Server → Backends → Add Backend,类型选 “ComfyUI API By URL”,逐个填地址:

后端名Address 填这个
A1…A6http://127.0.0.1:8188 … :8193(A 本机 6 个)
B1…B3http://SERVER_B:8188 … :8190(← 改成 B 的 IP)
模型放置 = 自动分流(关键技巧)。把视频模型(Wan2.2)只放在 A 的 ComfyUI 模型目录,把出图模型(Flux)只放在 B 的模型目录。这样“要 Wan 的任务”只能落到有 Wan 的 A 后端、“要 Flux 的任务”只能落到 B 后端——不用额外配路由,靠模型文件位置天然把视频钉在 H100、出图钉在 5090。

第 4 步:验证每张卡都在干活

分别在 A、B 上执行
nvidia-smi     # A 上应看到 GPU0/1 被 vLLM 占,GPU2~7 各一个 python(ComfyUI)
               # B 上应看到 GPU0/1/2 各一个 python,GPU3 被音频进程占
怎么算成功:SwarmUI 的 Backends 页 9 个后端全绿;Generate 页出图/生成视频都能出结果。每个后端一次只跑一个任务,9 个后端 = 9 个镜头同时渲染,其余排队。
更省事的替代法:SwarmUI 自启动后端(单机适用)

如果只用服务器 A,可以不手动起 ComfyUI:在 SwarmUI 里加 “ComfyUI Self-Starting” 后端,每个后端的设置里把 GPU_ID 填 2、3、4…由 SwarmUI 自己拉起进程并钉卡。跨机器(B)时仍建议用上面的“ComfyUI API By URL”远程登记法,更透明可控。

n8n导入可直接跑的流水线

下面两个文件是搭好的 n8n 工作流。下载 → 在 n8n 里导入 → 改几处地址就能跑,不用从零拖节点。

MAIN.json 主流程:①触发→②拆分镜→审核→④⑤渲染→⑥⑦音频→⑧合成→⑨通知 10 个节点 · 总调度 SUB_render.json 渲染心脏:一次性投递全部镜头 → 轮询做完几个 → 全好再继续 16 个节点 · 关键帧/视频通用
1

导入

n8n 右上角 ⋯ → Import from File,分别导入这两个 JSON。先导 SUB_render,再导 MAIN。

2

接上凭据 & 地址

  • Redis 节点:点任意 Redis 节点 → Credential → 填 SERVER_A:6379。
  • HTTP 节点:把 SERVER_A:7801(SwarmUI)、ntfy 通知地址改成你的真实 IP/端口。
  • 文件节点:确认路径都是共享盘 /media/...(n8n 容器要挂到同一个 /media)。
3

把子流程 ID 填回 MAIN

MAIN 里那几个 Execute Workflow 节点,当前写着 REPLACE_SUB_render_ID 之类占位符。点开每个节点,在下拉里选中你导入的对应子流程即可(SUB_parse / SUB_render / SUB_audio / SUB_assemble / SUB_qc)。

本次只给了 MAIN 和 SUB_render 两个;SUB_parse / SUB_audio / SUB_assemble / SUB_qc 结构更简单(基本就是几个 HTTP + Code 节点),可以照 SUB_render 的样子自己搭,或让我下一步一并补上。

SUB_render 的运转逻辑(导入后你会看到)

  • 展开镜头列表:读 project.json,摊平成“每个待渲染镜头一条”,已完成的自动跳过。
  • 逐镜投递:把每个镜头组装成 SwarmUI 载荷 POST 出去,只投递、不等结果——9 个后端在背后并行抽干。
  • 轮询循环:每 15 秒查一次 Redis 的 done 计数,没做完就继续等,全做完才往下走。
  • 回填 + 质检:把产物路径写回 project.json,交给 SUB_qc 逐镜打分,不合格的标 failed、下一轮只重做它。
导入后请自查两处:①Code 节点里的读写用的是 readWriteFile 节点,确认 n8n 有权限访问 /media;②done 计数由“ComfyUI 完成回调”写入 Redis 最稳——生产环境建议给每个后端配一个产物落盘后 INCR 的回调,而不是靠 n8n 猜。
