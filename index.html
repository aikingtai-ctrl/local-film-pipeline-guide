全本地影视流水线 · 从剧本到成片

一份给「小白」的搭建指南 —— 剧本丢进去,成片自动出来,全程在你自己的两台服务器上跑,不上云、不花 API 钱、素材不外泄。

🔒 全本地 数据不出门 🖥️ 8×H100 + 4×5090 双服务器 🤖 n8n 全自动编排 🎞️ 剧本 → 成片 一条龙

1这是什么?一句话讲清

你写好剧本,机器自动帮你拆分镜、画面、生成视频、配音、配乐、剪辑,最后吐出一部成片。

打个比方:一条自动化的「影视工厂流水线」。
剧本是「原材料」,从传送带一头放进去;沿途有 9 个「工位」各干一件事(切分镜、画图、生成视频、配音……); 最后一头出来的是「成品」——一部 mp4。

而 n8n 就是这条流水线的「总控制台」:它不亲自干活,而是按顺序指挥每个工位、传递半成品、出问题自动重做。

整套东西全部装在你自己的机器上。「本地」意味着:不联网也能跑、不给任何公司交钱、剧本和画面永远在你硬盘里。

412 张卡怎么分工

好消息:你不用手动管每张卡。SwarmUI 和 vLLM 会自动把活派给空闲的卡,你只要在安装时告诉它们“这几张卡归你管”。

机器卡干什么为什么这么分
A · 8×H1002 张大脑(vLLM)H100 显存大,能装下最聪明的大模型
6 张生成视频(主力)视频最吃资源,交给最强的卡并行猛跑
B · 4×50903 张画关键帧5090 画图飞快,适合大批量出图
1 张配音+配乐+字幕+美化这些活轻,一张卡全包
并行的威力:一集 150 个镜头的视频,单张卡要跑一整夜;6 张 H100 一起上,大约 1 小时就全出来了。两台服务器还能同时做两个不同的项目。

5动手搭建(照着敲)

下面每一步都有可复制的命令。点命令框右上角「复制」,粘到服务器终端回车即可。凡是写着 ← 改这里 的地方,换成你自己的信息(比如 IP、模型路径)。

开始前:两台服务器都要先装好 Docker 和 NVIDIA 显卡驱动 + NVIDIA Container Toolkit(让 Docker 能用显卡)。这一步通常运维已经装好;没装的话搜“NVIDIA Container Toolkit 安装”按官方文档走一遍。验证成功:运行 nvidia-smi 能看到你的卡列表。
1

装 n8n(总指挥台)+ Redis(小黑板)

服务器 A

这两个是流水线的“大脑指挥部”,不吃显卡,一起装:

在服务器 A 的终端执行
# 1) 启动 Redis(记录每个镜头的进度)
docker run -d --name redis -p 6379:6379 --restart always redis

# 2) 启动 n8n(总指挥台,网页操作)
docker run -d --name n8n -p 5678:5678 --restart always \
  -e GENERIC_TIMEZONE="Asia/Shanghai" \
  -v n8n_data:/home/node/.n8n \
  docker.n8n.io/n8nio/n8n
怎么算成功:浏览器打开 http://服务器A的IP:5678,看到 n8n 的注册页面,设个账号密码就进去了。这就是你以后搭流水线、点“运行”的地方。
2

装 vLLM(大脑:读剧本 / 打分)

服务器 A · 2×H100

先把大模型下载到本地(比如放 ~/models),再用 Docker 起服务。它会占用第 0、1 号两张 H100:

在服务器 A 的终端执行
docker run -d --name vllm --restart always \
  --gpus '"device=0,1"' \                    # ← 用第 0、1 号 H100
  -p 8000:8000 \
  -v ~/models:/models \                       # ← 改这里:你放模型的文件夹
  vllm/vllm-openai:latest \
  --model Qwen/Qwen2.5-72B-Instruct \         # ← 大脑模型(读剧本)
  --tensor-parallel-size 2                     # 两张卡合力跑一个大模型
质检打分(第⑨步给镜头挑毛病)要用“会看图的模型”,可以再起一个 Qwen2.5-VL 服务,或换成能看图的版本。先跑通文字版即可,看图的以后再加。
怎么算成功:执行 curl http://localhost:8000/v1/models 能返回模型名字,说明大脑上线了。
3

装 SwarmUI(画面工厂,管 12 张卡)

A 主控 + B 帮手

SwarmUI 是所有“画图 / 生成视频”任务的总管。你在它的网页里,把每张卡登记成一个“后端”,它就会自动把活派给空闲的卡。

在服务器 A 上安装 SwarmUI(Linux)
git clone https://github.com/mcmonkeyprojects/SwarmUI
cd SwarmUI
./launch-linux.sh            # 首次运行会自动装依赖,按提示走完

装好后浏览器开 http://服务器A的IP:7801,进 Server → Backends 页面,按下面思路登记后端:

  • 服务器 A 上加 6 个 ComfyUI 后端,分别绑 H100 的第 2~7 号卡(用 CUDA_VISIBLE_DEVICES=2、=3…区分)→ 专做视频
  • 服务器 B 上再装一份 SwarmUI/ComfyUI 后端,加 3 个绑 5090 → 专做关键帧,登记到 A 的 SwarmUI 里当“远程帮手”
这一步是全流程最需要耐心的地方(要一张张登记卡)。可以先只登记 1~2 张卡跑通整条流水线,验证没问题后再把剩下的卡补上。具体每张卡的绑定参数,我可以在下一步「C」里给你一份逐卡配置表。
怎么算成功:在 SwarmUI 的 Generate 页随便输入一句英文描述点生成,能出图,说明画面工厂通了。
4

装声音三件套(配音 / 配乐 / 字幕)

服务器 B · 1×5090

这三个都比较轻,共用 B 的第 4 张 5090。装好后各自开一个网络接口,n8n 到时候调用它们:

软件作用怎么装(思路)
IndexTTS2配音/旁白(可克隆音色)按其官方仓库装,用 FastAPI 包一个 /tts 接口
ACE-Step按情绪生成背景音乐官方仓库,包一个生成接口
faster-whisper把配音转成字幕、对齐时间官方仓库,包一个 /transcribe 接口
这三个的目标是:各自能通过一个网址被调用(比如 http://服务器B:9880/tts)。只要能用浏览器或 curl 调出结果,n8n 就能接上。
5

建共享硬盘(两台机器看同一个文件夹)

A 和 B

A 生成的视频、B 生成的图,必须让两台机器都能看到。做法:在 A 上建一个共享文件夹(NFS),B 挂载它。这样两台机器眼里都是同一个 /media。

A 当“硬盘提供方”,B 当“借用方”(示意)
# 服务器 A:把 /media 共享出去(装 nfs-kernel-server 后)
echo "/media  服务器B的IP(rw,sync,no_subtree_check)" | sudo tee -a /etc/exports
sudo exportfs -ra

# 服务器 B:挂载 A 的 /media 到本地同名目录
sudo mount 服务器A的IP:/media /media       # ← 改这里两个 IP
流水线里所有素材都放这个共享 /media,n8n 传递的永远是“文件路径”(比如 /media/项目名/clips/xxx.mp4),不要在软件之间传图片本身,否则内存会爆。
6

在 n8n 里把流水线搭起来

浏览器 · n8n 网页

回到 http://服务器A:5678。流水线由 1 个主流程 + 5 个子流程组成,像搭积木一样在网页里拖节点、连线。你先建这几个工作流:

工作流负责核心节点(在 n8n 里搜这些名字)
MAIN 主流程总调度,按①→⑨顺序指挥Webhook、Execute Workflow、Wait、HTTP Request
拆分镜第②步,调 vLLMHTTP Request(打 vLLM)、Code
画面渲染第④⑤步,调 SwarmUI(关键帧和视频共用)Split in Batches、HTTP、Wait、IF、Redis
质检第⑨步,让大脑给镜头挑毛病、决定重做HTTP(打 VLM)、IF
配音配乐第⑥⑦步HTTP(打声音三件套)
合成第⑧步,ffmpeg 剪辑Execute Command(跑 ffmpeg)
不用手写代码从零搭。核心的“画面渲染”流程有固定套路:一次性把所有镜头丢给 SwarmUI → 它自动分给 12 张卡 → n8n 每 15 秒查一次“做完几个了” → 全做完再进下一步。这套节点结构和里面的配置,我可以直接给你一份能导入 n8n 的文件,你导进去改改地址就能用。
怎么算成功:在 MAIN 流程点“Test workflow”,丢一小段剧本,能看到节点一个个变绿、最后 /media 里出现成片 mp4,整条线就通了。

6搭好之后,平时怎么用

  1. 把写好的剧本存成 .txt,放进监听文件夹(比如 /media/inbox/)。
  2. 流水线自动开工,或你在 n8n 点一下“运行”。
  3. 到了“留人工过目”的节点(分镜、角色图),你手机/电脑会收到通知,点开看一眼,满意就点“通过”。
  4. 等 1~2 小时,去 /media/项目名/out/ 拿成片。质量差的镜头系统已自动重做过。
两台服务器 12 张卡,可以同时推进两个项目——比如横屏纪录片和竖屏短剧一起产。

7核心文件 project.json:整条线的“施工总表”

整条流水线只围着一个文件转:project.json。你可以把它想成“一张不断被填写的施工总表”。

第②步拆分镜时,大脑先把它写成一张空白总表:有多少场戏、每场几个镜头、每个镜头画什么、说什么、多长、什么情绪——但“画好的图”“生成的视频”这些格子先空着。后面每个工位干完活,就把自己的成果填进对应的空格:

project.json 里一个“镜头”长这样(简化)
{
  "id": "s01_sh01",              // 镜头编号:第1场第1个镜头
  "duration": 4.0,              // 时长(秒)
  "shot_size": "WS",           // 景别:远景
  "camera": "push_in",         // 运镜:缓慢推近
  "prompt_visual": "昏暗地下宫殿,金柱倒影...",  // 画面描述
  "narration": "这里……本不该存在。",           // 旁白
  "mood": "悬疑压抑",           // 情绪(决定配什么音乐)

  "keyframe": null,            // ← 第④步填:画好的图
  "clip": null,                // ← 第⑤步填:生成的视频
  "vo": null,                  // ← 第⑥步填:配音文件
  "status": "pending"          // 进度:等待中→排队→已生成→通过
}
为什么这样设计?因为每个镜头都单独记着“做到哪一步了”。万一第 87 号镜头视频没生成好,系统只重做这一个,不用把整集重来。这就是流水线又快又稳的秘密。

8常见问题 & 排错

Q:我完全没写过代码,能搭起来吗?

能,但需要耐心。第 5 步的命令都是“复制粘贴回车”。真正需要动脑的是 n8n 里拖节点连线(第 5.6 步)和 SwarmUI 登记显卡(第 5.3 步)——这两块我可以给你更细的“保姆级”配置文件和图示。建议:先只用 1~2 张卡把整条线跑通,体验一遍全流程,再扩到 12 张卡。

Q:视频生成看起来很“廉价”怎么办?

三招提质量:①给主角训练专属形象(LoRA),全片不换脸;②每个镜头生成 3~4 个版本,自动挑最好的;③生成后用补帧+超分把画面美化到 1080P/60 帧。你的硬件很强,这些“奢侈操作”几乎不额外花时间。

Q:某个软件起不来 / 网页打不开?

按顺序查:①docker ps 看容器是不是在运行;②docker logs 容器名 看报错;③防火墙有没有放行端口(5678/8000/7801 等);④显卡类的看 nvidia-smi 卡在不在、有没有被别的进程占满显存。

Q:一定要两台服务器都用上吗?

不。可以先只用服务器 A 一台跑通所有环节(A 的卡够强,全包没问题)。等熟练了再把 B 接进来分担画图和配音,提升整体速度。

Q:剧本拆分镜拆得不好怎么办?

调第②步给大脑的“指令词”(prompt)和输出规则。把一段真实剧本喂进去看结果,不满意就细化要求(比如“每个镜头别超过 5 秒”“对白单独放一格”)。这是可以反复打磨的。

9名词对照表(看不懂就查这)

Docker绿色安装器。一条命令装好一个软件,互不干扰,删了也干净。

prompt给 AI 的“指令词/描述”,比如画面要画什么、剧本怎么拆。
